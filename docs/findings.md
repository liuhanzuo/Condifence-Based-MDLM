## 核心问题

一个未揭示位置 i 在部分揭示状态 s 下的"跨轨迹后验稳定性" Stability(i|s)：
- **A 存在性**：是否独立于 predicted confidence？
- **B state-local 可判定性**：能否只由 hidden state h_i(s) 预测（不看全局揭示集 R）？
- **C 可诱导性**：能否训练目标增大 commit-ready 比例？（未做）

Stability(i|s) = Pr_τ[ argmax p(x_i|s') == argmax p(x_i|s) ]，s' 为从 s 出发随机揭示其余位置到不同程度的中间态。

## 已完成实验与结论

### Exp1-3（命题 A/B + 进度-耦合拆解）✅ 完成
数据：1570 样本，20 句 × 12 anchor × K=12，keep_ratio∈[0.05,0.7]

- **命题 A 成立**：Spearman(prob, stability)=0.32 → readiness 独立于 confidence。
- **命题 B 基本正向**：
  - probe0 [prob,margin] R²=0.114
  - probe1 [hidden, state-local] R²=0.340  ← hidden 单独预测能力是 confidence 的 3 倍
  - probe3 [hidden + 全局R] R²=0.402  ← 加全局信息仅 +0.06 → readiness 主要是 state-local
- **发现一个真实的早期 path-dependence**：同进度分层后，
  - nrev∈[1,2]: Δlocal=+0.092（极稀疏时"谁被揭示"明显重要）
  - nrev∈[2,5]: Δlocal=+0.030（变弱）
  - nrev∈[5,9]: stability 饱和≈1
  → 结论：readiness 大体 state-local，**除生成最早期**；早期耦合 = 语义单元边界尚未建立。
  → 机制上解释了 block-wise 为何有效（先建锚点，再块内并行）。

## Exp5 结果：逐层 probe（readiness 编码在哪一层）✅

1321 样本，扫 15 个层（0=embedding, 28=最后）：

- layer 0 (embedding): R^2=-0.00（无信息，合理）
- **layer 2: R^2=+0.379（最强）**，layer 4: 0.359
- layer 6-12: 掉到 0.13-0.23（中层反而弱）
- layer 14-26: 回升并稳定在 0.31-0.34
- baseline [prob,margin]: 0.099

**发现**：readiness 信息呈"早层强→中层弱→后层回升"的 U 形，**浅层(第2层)最强**。
→ 提示 commit-readiness 更像一种**浅层/局部的表面可预测性统计**，而非深层语义。
→ 对 certifier 设计有利：可能用浅层特征即可判 readiness，成本低（不必跑完整网络）。
⚠️ 第2层最强可能含位置/token统计的 artifact，标记为待复核观察，不夸大。

## Exp6 结果：AR 退化的直接测量 ✅

让模型 confidence-first 自发揭示（20 句），测揭示顺序：

- **揭示顺序 vs left-to-right Spearman = +0.62 ± 0.13** → 强偏 AR 但非纯 AR（纯AR=+1）
- **相邻揭示间距=+1 的比例 = 0.53** → 一半以上是局部 left-to-right
- **典型 order 模式**：`[0,1,2, 11(句尾), 3,4, 9,7,10,8, 5,6]`
  → 先锚定开头连续几个 token + 句尾，再在锚点间填充。

**解读（重要）**：这不是纯 token-wise AR，而是"**先建结构锚点(开头+结尾)，再锚点间填充**"的模式。
这正是 information-wise 生成的经验证据，支持"diffusion 体现在锚点/单元层面，局部填充呈 AR"的假设，
也与 Exp1-3 发现的"早期 path-dependence 集中在锚点建立阶段"一致。

## Exp7 结果：certifier 实用价值（最强正向结果）✅

场景：并行 commit top-r 比例的位置，比较用 confidence / certifier(读hidden) / oracle 选择，
指标 = 被 commit 位置的平均真实 stability（越高=commit越安全）。1321 样本, 5 折 CV。

| commit 比例 | random | confidence | **certifier** | oracle |
|---|---|---|---|---|
| 0.1 | 0.849 | 0.997 | **0.999** | 1.000 |
| 0.2 | 0.876 | 0.985 | **0.998** | 1.000 |
| 0.3 | 0.860 | 0.978 | **0.996** | 1.000 |
| 0.4 | 0.878 | 0.966 | **0.994** | 1.000 |
| 0.5 | 0.897 | 0.948 | **0.990** | 1.000 |

**结论**：certifier 在所有并行度下都优于 confidence，且**并行度越高优势越大**。
- 并行 commit 50% 时：错误率从 confidence 的 5.2% 降到 certifier 的 1.0%（**降低约 80%**）。
- certifier 几乎贴合 oracle 上界。
→ 直接证明"读 hidden state 判断 commit-readiness"比"用置信度阈值"在并行生成场景实质更优，
  这正是整条研究线的核心价值主张，且优势集中在最需要并行的激进场景。

## Exp8 结果：锚点是位置性还是语义性？（反证 information-unit 假说）⚠️

对自发揭示顺序，回归"揭示早晚 vs 位置/语义特征"（225 tokens）：

- corr(揭示步序, is_edge)=-0.19，corr(step, center_dist)=-0.19 → **靠边(首尾)的位置更早揭示**
- corr(揭示步序, is_content)=+0.04；控制位置后内容词偏系数≈-0.02 → **内容词几乎无独立效应**
- 内容词平均步序 0.509 vs 功能词 0.451 → 功能词反而略早

**结论**：Exp6 的"锚点优先"主要是**位置性的（先填句子首尾边界）**，
**不是语义性的**（不是先填信息量大的内容词）。
→ 这是对 GPT "information-unit（先建语义核心）"叙事的**反证据**——至少此模型的自发顺序由位置驱动。
→ 提醒：别被优雅叙事带走，数据支持的是"边界锚定"而非"语义单元优先"。

## Exp9 结果：不可逆代价 + AR 退化的 confidence 门控机制 ✅

按 confidence≥0.9 才 commit（1321 样本）：只有 3.6% 的位置达标，且**几乎全在充分揭示之后**：
- 早(nrev≤2): 无 commit；中: 3 个；晚(nrev≥4): 44 个，错误率均 0。

**关键机制发现**：`confidence 高` 这件事本身 ≈ `上下文已充分`。因此 confidence 门控天然强制：
> 要 confidence 够高才敢 commit → confidence 只在邻居揭示后才够高 → 只能等上下文充分再 commit
> → 退化成"上下文优先"的串行 = AR。

→ 这给"为什么退化成 AR"一个干净的定量解释：**不是语言是 AR，而是 confidence 门控强制了串行**。
→ 也正是 certifier(Exp7) 的价值所在：它能在 confidence 还不高时就识别 stable 位置，从而打破串行、支持并行。
（注：初版 Exp9"完整句重预测测后悔"因同模型同上下文自洽=循环论证，已废弃改为本版。）

## Exp10 结果：真实并行生成对比（补 Exp7 漏洞）+ 对"是否只是 planner"的诚实回答 ✅⚠️

设置：两种策略**各自真实生成整句**（confidence-topk vs certifier-topk 挑位置敲定），
16 个 prompt，gen_len=12，用**独立裁判 GPT-2** 测生成 NLL（越低越好）。扫每步敲定数 k。

| k(每步敲定) | 步数 | confidence NLL | certifier NLL | winner |
|---|---|---|---|---|
| 1 (≈AR) | 12 | 7.507 | **5.249** | certifier（差距最大）|
| 2 | 6 | 6.461 | **5.399** | certifier |
| 3 | 4 | 6.641 | **6.129** | certifier |
| 4 | 3 | 6.099 | **6.012** | certifier |
| 6 | 2 | 6.210 | **6.142** | certifier |
| 12 (全并行一步) | 1 | 6.016 | 6.016 | 相同(无排序动作,sanity check) |

**正面**：certifier 在真实生成里**每个并行度都赢或平**，补上了 Exp7"没真并行"的漏洞。
它是一个**明显更好的 planner**（k=1 时 NLL 7.5→5.2，非微弱优势）。

**关键诚实结论（回答"是否只是换了个 planner"）**：
- certifier 优势**在低并行(k小,串行)时最大、高并行时迅速与 confidence 趋同**（与 Exp7 趋势相反）。
- 说明当前收益主要来自"它是一个更好的**揭示顺序 planner**"，**不是**来自"解锁高并行不退化"。
- → **确认了 vanilla/planner 的自我质疑：当前形态本质就是 planner 谱系里"更好的评分函数"**。
- → 高并行区间(k≥4)两条曲线趋同，正是"待真正解决"的地方 = 命题 C 的靶子。

## 命题 C：训练诱导 commit-ready 表征（进行中）

**目标**：跳出"inference-only planner"。加一个辅助训练目标让模型 hidden 显式编码 stability，
看能否把 Exp10 高并行区间(k≥4)的生成质量压到 inference-only certifier 之下
（这是任何 inference-time planner 做不到的，若成立=范式级证据）。

**最小设计（单卡约束）**：不全量微调 7B；用 LoRA(q/v_proj, r=8, 2.5M参数, 0.033%) + 辅助 readiness 头，
辅助损失 = 让第4层 hidden 预测该位置 stability(蒙特卡洛标签) 的 MSE；保 LM 去噪损失防语言退化。训练 150 步。
**判定**：高并行生成质量能否超过 inference-only certifier，且增益需归因于 readiness 表征(非纯微调)。

### 命题 C 结果：证据【不成立】（含关键对照）⚠️

真实生成 + 独立 GPT-2 裁判，评估 prompt 与训练句子不同。高并行区间(k≥4) 平均 NLL：

| 策略 | NLL(越低越好) |
|---|---|
| inference-only certifier (无训练) | 6.077 |
| **LoRA微调 + 普通confidence planner（对照）** | **4.539** |
| LoRA微调 + readiness planner (命题C) | 4.755 |

**关键**：初看 trainedC(4.76) 远超 certifier(6.08)，像是命题C成立；
但加入对照 **lora+conf(4.54)** 后真相反转：
- **增益几乎全来自 LoRA 微调本身**（大概率在20句小数据上过拟合，GPT-2觉得更流畅）。
- readiness planner 相对纯微调对照 **Δ=+0.216（反而略差）**，可能因微调改变hidden分布使readiness head OOD。

**结论**：这个最小版**没有证实**"训练诱导 commit-ready 表征"能带来 inference-time planner 给不了的收益。
增益来自微调副作用，而非目标机制。命题 C 在当前设计下**证据不成立**。

**为什么可能失败 / 后续该怎么改**：
1. 20 句训练数据太小，LoRA 过拟合主导了信号，淹没了 readiness 目标的效应。
2. readiness head 读固定层，微调后该层分布漂移 → head 与主干不同步（应 head 与 LoRA 联合适配，或 head 也随分布更新）。
3. 辅助损失只"预测"stability，没有真正"改变生成决策的可微路径"——表征被诱导了，但没接到生成动作上。
4. 评估指标(GPT-2 NLL)对"流畅度"敏感，微调过拟合会虚假拉低它，需换下游正确率类指标。
→ 命题 C 要成立，需更大数据 + 让 readiness 以可微方式真正参与 unmask 决策(而非仅辅助预测)。

### 命题 C 完整版（第二版）：可微 commit 驱动 + 更大数据，仍【不成立】且失败方式反转 ⚠️

针对最小版根因改进：(1) 训练数据 20→50 句；(2) 训练 150→400 步；
(3) **核心改动**：readiness 通过 straight-through soft-commit 可微地驱动"一步并行 commit"，
损失=commit 后剩余位置的 LM 预测损失（让 readiness 学"选出 commit 后整体更好预测的位置"），
而非仅旁路预测 stability。

高并行区间(k≥4) 平均 NLL：

| 策略 | 最小版 | **完整版** |
|---|---|---|
| inference-only certifier(无训练) | 6.077 | **6.077（最好）** |
| LoRA+confidence(纯微调对照) | 4.539 | 6.746 |
| LoRA+readiness(命题C) | 4.755 | 6.812 |
| Δ(readiness净增益) | +0.216 | +0.066 |

**两个关键发现**：
1. readiness planner 相对纯微调对照**仍无净贡献**（Δ=+0.066，略差）。
2. **失败方式反转**：可微 commit 损失把模型整体**搞差了**（lora+conf 从 4.54 退到 6.75），
   连完全没训练的 infer_cert(6.08) 都反超了它。推测：soft-commit 训练信号鼓励"一次填对一堆"，
   与真实语言建模冲突，把 LoRA 拉向"适合并行但语言质量更差"的方向。

**合并结论（命题C 两版）**：在单卡+小数据+LoRA 条件下，"训练诱导 commit-ready 表征"
**无法带来超越 inference-only certifier 的收益**：辅助损失版增益全来自微调副作用；
可微 commit 版反而有害。**readiness 信号始终未能独立转化为生成质量优势**。
→ 本质判断：commit-readiness 作为 **inference-time 判据有效**(Exp7/10)；
   但**塞进训练目标重塑表征**，在此规模下无效/有害。这不是调超参能解决的，指向更本质的困难。
→ 若要继续：需 (a) 显著更大数据与算力(超出单卡)；(b) 更温和地把 readiness 接入生成
   (如仅在推理调度用，训练只做轻正则)；(c) 换真实下游任务指标而非 GPT-2 NLL。

- **TraceLock (The Path Matters, 2605.24697)**：用 hidden state(+conf, 过AE降维)学"该token是否=最终值"→lock commit。

- **What-Unmasked-First (2605.31564)**：轨迹分析(content-first, 结构token当锚点) + λ-scaling，图到文本专用。

**关键共性**：以上全部 **inference-time / 冻结底座**——它们**检测**底座里"碰巧独立"的位置，
**没有一篇训练底座去主动增加独立结构**。→ 真空区 = "训练主动塑造条件独立结构"。

## 相关工作补充（第二轮 deep research）：planner/commit-order 新论文 + 一个对我们方法的关键警示 ⚠️

第二轮调研聚焦"训练时联合学习顺序"与"推理时冻结底座policy"两条线：

- **OeMDM/LoMDM (2602.02112, KAIST)**：顺序scheduler与主干**从零联合训练**(单一NELBO目标)，
  LM1B/OWT上PPL创SOTA、训练效率5.5x，但这是学"顺序"，不是像我们直接对pairwise KL依赖施加损失。
- **MDMs are Secretly Learned-Order AR Models (2511.19152)**：理论证明解码顺序↔噪声调度↔加权AR损失分解，
  但只在表格数据验证，**验证损失下降但下游生成质量提升有限**——与我们Exp18机制指标好看但下游质量不够硬是同一类陷阱。
- **UPO (2510.05725, KAIST)**：KL正则化MDP+GRPO训练轻量policy(冻结LLaDA-8B)，有严格收敛理论证明；
  证明"训练policy"与"训练底座(diffu-GRPO)"**可叠加互补**。
- **DCoLT (2505.10446, NeurIPS'25)**：**联合**RL训练unmasking policy(Plackett-Luce)+底座本身，
  GSM8K+9.8%/HumanEval+19.5%，效果比只训policy更猛（但要动底座，算力需求16×H800）。
- **Where-to-Unmask (2602.09501)**：定义Gt-Margin(GT token概率与次高候选margin)构造oracle顺序，
  learning-to-rank监督训练planner模仿，底座冻结。
- **Train for the Worst, Plan for the Best (2502.06768, ICML'25, Kakade组)**：理论证明MDM训练时
  被迫学习比AR更难的指数多"填空"子问题，但推理时自适应选序可规避难题，Sudoku从<7%→~90%。

**判决（同第一轮）**：以上全部聚焦"学习/选择解码顺序"，**没有一篇直接对pairwise conditional dependency
本身施加训练损失**——命题C-v3"精准解耦"的角度依然是真空。

### ⚠️ 关键警示：The Flexibility Trap (2601.15165, ICML'26 Oral, 清华+阿里, JustGRPO)

**核心发现**：dLLM的任意顺序生成在数学/代码推理任务上**不是优势而是陷阱**——
"easy-first"式的unmask策略会让模型**绕开高熵的逻辑分叉token**（如"therefore/thus/since"），
等这些token周围的上下文都填完后再回填，此时分支的不确定性已被消除，
**分叉token不再是真正的branching point，导致解空间(pass@k)过早坍缩("entropy degradation")**。
解法JustGRPO：RL训练阶段放弃任意顺序，把dLLM当AR policy算标准GRPO（只改训练目标计算方式，
不改模型结构、不加causal mask），推理时仍保留并行解码。GSM8K上比多个diffusion专用RL方法更强
（89.1 vs ESPO 82.3/GDPO 82.8/SPG 86.1）。

**对我们方法的直接警示**：命题C-v3的解耦损失 = 无差别压低"当前最强的" pairwise KL依赖，
**只看依赖强度，不看这个依赖是否是推理所需的"分叉耦合"**。GSM8K推理链里的连接词/关系词
对上下文的强依赖很可能正是承载分支决策的**必要耦合**，而不是"冗余"——我们的算法可能恰好在
系统性地打掉这类依赖，这比Flexibility Trap描述的"绕开"更激进(直接从表征层面抹掉耦合)。
**这给出了一个新的机制假说**，可以部分解释为什么Exp18的GSM8K下游质量始终没起色：
不只是训练格式/OOD问题，解耦目标本身在推理类任务上可能方向有误。

**待验证的诊断实验**：检查命题C-v3训练时被选中解耦的强依赖pair，是否显著落在
"therefore/so/thus/算式关系词"等分叉token上（vs 落在真正冗余的表面依赖上，如重复的
限定词/介词）。若前者占比高，则证实了这个警示，需要把解耦范围**排除掉fork token**，
只打真正语义冗余的依赖；若后者占比高，说明我们的方法本身没踩这个陷阱，问题另有原因（如训练规模不足）。

### Exp19 诊断结果：假说被部分证伪，但发现更具体的问题——解耦在打"数字间依赖" ⚠️

用 `experiments/proposition_c/19_diagnose_fork_tokens.py`，在58个GSM8K训练样本上跑一遍
"找强依赖pair"逻辑(不训练)，统计167个被算法选中要解耦的pair、334个token端点的类别分布：

| 类别 | 占比 | 例子 |
|---|---|---|
| 数字 | **34.7%** | 2, 2, 2, 0, 5, 5, 9, 4 |
| 其他内容词 | 20.1% | fish, males, dollars, cards |
| 限定词/介词等短词 | 16.8% | of, age, is, The |
| 运算符/符号 | **13.5%** | =, -, /, $ |
| 空白/换行 | 11.4% | — |
| **fork/连接词** | **3.6%** | So, because, if, Then |

**判读**：
1. **原始假说（打的是fork/连接词）证据不强**——fork/connective只占3.6%，Flexibility Trap
   描述的"绕开逻辑分叉token"机制在我们的解耦目标里**不是主要作用对象**。
2. **但发现了一个更具体、可能更直接有害的问题**：数字+运算符/符号合计占比 **~48.2%**，
   是被解耦算法打击最多的类别。这在GSM8K这种算术推理任务上是**直接有害的**——
   "5+3="后面的"8"天然应该强依赖于"5"和"3"，这不是冗余噪声，是算术正确性的必要条件。
   我们的解耦损失在系统性地压低数字/运算符之间的依赖，**可能在教模型"预测数字时更少依赖
   上下文里的其他数字"**，这与算术推理的需求直接冲突。
3. **两版假说的共同教训**：无论是"打掉逻辑连接词"还是"打掉数字间依赖"，本质问题相同——
   **命题C-v3的"哪个依赖最强就打哪个"选择规则，没有区分"语义冗余的依赖"和"任务正确性
   所必需的依赖"**。在declarative短句/wiki语料上这个问题不明显(冗余依赖占主导)，但在
   GSM8K数学推理上，强依赖pair里近一半是数字/运算符——这类任务恰好放大了这个设计缺陷。

**结论与下一步**：这为命题C-v4指出一个具体、可操作的修正方向——**在强依赖pair候选中
排除数字-数字、数字-运算符pair**（以及更广义的、可用简单规则或POS/NER识别的"计算依赖"），
只对真正冗余的表面依赖（如限定词/介词/重复修饰词，测试中占比合计~37%）施加解耦损失。
这比原Flexibility Trap假说更贴合我们的具体场景，值得在H100上优先验证。

## Exp13 诊断：天然条件独立子集有多大？（判决"训练增加独立性"是否有空间）✅



用 pairwise dependency（factorization gap 成对近似）：dep(i,j)=KL(p(x_i|s) ‖ p(x_i|s+reveal j))。
1701 个 pair，64 个部分揭示状态：

- **pairwise dependency 中位数=0.77，均值=1.54；dep>0.3(强依赖)的 pair 占 69.3%**。
- **天然可并行独立子集占比**：τ=0.05→16%，τ=0.1→23%，τ=0.3→32%。
- **confidence top-k 选中的位置里，63% 的 pair 其实强依赖**（confidence 会把强依赖位置一起并行→出错）。

**判决**：
1. **天然条件独立的位置很稀疏**（宽松阈值也只 ~32% 可并行，69% 强依赖）
   → DEMASK/PUNT 这类"检测天然独立"的方法**天花板受限于底座本身独立结构稀疏**。
   → **"训练主动增加条件独立结构"有真实的大片空间（那 68% 强依赖里争取）** = 真空方向可做。
2. confidence 抓不住依赖（63% 冲突）→ 再次印证 confidence 非并行判据（与 Exp1/9 一致）。
3. **对命题C的修正启示**：命题C失败或因无差别对所有位置压 readiness；
   更精准的目标是**只针对"本可独立但模型未学会独立"的强依赖 pair**做训练，而非全局压。

**当前最有希望的方向**：不是"检测独立性"（已被 DEMASK/PUNT 占），而是
**"训练让模型对本不必依赖的 token 对变得条件独立"**——这是所有现有工作的公共盲区，
且 Exp13 证明存在 ~68% 的可争取空间。风险仍是命题C揭示的"独立性 vs 语言建模"张力，
但可用"只针对冗余强依赖 pair"的精准目标规避。

## 命题 C 第三版：精准解耦（Targeted Decoupling）——首次出现正向机制信号 ✅（待补对照）

设计：只对"当前强依赖(dep>0.3)"的 masked pair 施加解耦损失 = 降低 dep(i→j)=KL(p(x_i|s)‖p(x_i|s+reveal j))，
同时保 LM 损失护栏。LoRA(q/v) 训练 300 步。（区别于前两版：直接用 factorization gap 做训练信号，且只打强依赖 pair）

**训练**：lm 9.4→0.09（未崩），解耦目标 dep 3.86→0.93（降 76%）。

**评估（训练集外句子/prompt）**：
- **(A) 机制指标**：dep 中位数 0.65→0.15；强依赖(dep>0.3) 占比 **69%→28%**；
  天然可并行子集(τ=0.3) **32%→64%（翻倍）**。→ **训练确实增加了条件独立结构，且泛化到训练集外**。
- **(B) 高并行生成质量**：base(conf)=6.46 → decoupled(conf)=4.30，Δ=-2.16。

**这是三版里首个正向机制信号**：前两版机制指标从没动过，第三版让 dependency 真正大幅下降且可泛化。
**但生成质量(B)的对照尚不干净**：base vs decoupled 两者都用 confidence planner，但 decoupled 是微调模型，
质量提升可能混入"LoRA 微调在小数据上的过拟合副作用"（lm→0.09 提示过拟合）。
**待补关键对照**：纯 LM 微调(不加解耦损失)的模型——若它不降 dep，则机制结论干净；若也降 dep，则打折。
（机制指标 A 因 dep 是解耦损失的直接目标，纯微调理论上不该专门降它，故 A 大概率是解耦真贡献，但需对照坐实。）

### 命题 C 第三版·判决性对照：解耦 vs 纯LM微调（lam_dec=0）——机制结论坐实 ✅✅

补了关键对照：同样 LoRA、同样 pair 采样流程，但关掉解耦损失(lam_dec=0)=纯 LM 微调。均训练集外评估。

| 指标 | 训练前 | 纯LM微调 | 解耦 |
|---|---|---|---|
| dep 中位数 | 0.646 | **1.152(↑变差)** | **0.152(↓大降)** |
| 强依赖pair占比 | 68.6% | **76.2%(↑)** | **27.7%(↓)** |
| 独立子集 τ=0.3 | 32.4% | 34.8%(≈不变) | **63.9%(翻倍)** |
| 高并行NLL(k≥3) | 6.456(base) | 4.710 | 4.295 |

**决定性结论**：
- **纯 LM 微调不降反升 dependency**（0.65→1.15）；**只有加解耦损失才大降**（0.65→0.15，可并行子集翻倍）。
  → dependency 下降/独立结构增加是**解耦损失的真实贡献，非微调副作用**（排除了命题C第一版的陷阱）。
  → **这是三版命题C里首个通过严格对照的正向机制结果**。
- 生成质量：纯微调 Δ=-1.75，解耦 Δ=-2.16 → 解耦净贡献约 **-0.4**（小而正），大头仍是微调副作用，此项尚不够硬。

**综合**：命题 C 在"精准解耦"形态下**部分成立**：
① 机制上——训练能主动、干净、可泛化地增加条件独立结构（强依赖 69%→28%，可并行子集翻倍），
   这是所有现有 inference-only 工作（DEMASK/PUNT）做不到的（它们只能检测天然独立）；
② 生成质量上——解耦有小净正贡献但被微调噪声主导，需更大数据/更强评估坐实。
→ 这首次把"training distribution / 主动塑造条件独立结构"从设想变成了**有对照支撑的正向证据**，
   且正好落在文献真空区。是目前整条线最有价值、最可能成文的结果。

**下一步（坐实②）**：更大训练数据(减过拟合) + 真实下游任务指标(替代GPT-2 NLL) + 与DEMASK同台比较
（相同并行度下，"解耦训练+简单planner" vs "冻结底座+DEMASK检测"，看训练增加独立性是否超过检测天然独立的上限）。

## 命题 C 第三版：能说明 / 不能说明 什么（分寸）

**能说明（有对照，站得住）**：
1. **条件独立结构可训练**：一个只打"冗余强依赖对"的解耦目标，能主动、大幅、可泛化地降低 token 间条件依赖
   （强依赖 69%→28%，可并行子集 32%→64%），且纯LM微调对照不降反升 → 是解耦的因果贡献。
2. **揭示并证伪了主流路线的天花板**：DEMASK/PUNT 等只"检测"天然独立(仅~32%)；
   本实验证明该天花板是"没人训练它"而非语言本身 → "训练制造独立"是与 inference-only 正交的真空方向且有东西。

**不能说明（必须诚实）**：
1. **未证明"降依赖→并行生成质量更好"**：质量(NLL)上解耦净贡献仅~-0.4，大头是微调副作用，下游链条后半段无硬证据。
2. **未排除"降依赖是否损害语言能力"**：lm损失过拟合(→0.09)，没在大规模独立语料测语言建模能力是否退化（命题C二版正是栽在此）。
3. **规模有限**：50句/单模型(Dream-7B)/成对KL近似 → 强 proof-of-concept，非定论。

**一句话**：证明了因果链前半段（依赖可训练降低，且正交于主流、落在真空区）；
后半段（降依赖能否换来更好/更快/不损害语言的并行）尚无硬证据。
价值 = "找到并验证了一扇没人走、且前半段走得通的门"，而非"解决了并行生成"。

## Exp18：GSM8K 真实下游任务验证——checkpoint 严重 OOD，评估不可用 ⚠️（负向发现）

目标：换掉 GPT-2 NLL 代理指标，用 GSM8K（1319题测试集，HF `openai/gsm8k`，经 hf-mirror 下载）
真实数学题正确率验证命题C第三版"降依赖→并行生成更好"的因果链后半段。

设计：base(冻结) / lmonly(纯LM微调) / decoupled(精准解耦) 三模型，chat模板+system prompt
引导输出`#### <number>`，在 k∈{2,8,24/40}并行度下生成，规则抽取答案与GT做exact match。

**结果（gen_len=48 与 256 两次独立复现一致）**：
- **base**：部分题目能正确推理并输出答案（k=8时 4题中2题对，含合理中间过程）。
- **lmonly 与 decoupled**：**几乎全部退化成重复token**（如`"the the the..."`、`"books books..."`），
  无法产生连贯推理，正确率≈0，且两者退化程度相近（不是解耦独有问题）。

**诊断**：lmonly/decoupled 用的训练数据是当时沿用的 `data_pool.TRAIN_SENTS`——
仅 50 句极简短陈述句（8-50 token，keep_ratio 0.15-0.5，无 chat 格式/无推理链）。
300步 LoRA 微调后**严重过拟合到这个窄分布**，导致在 GSM8K 这种更长、chat模板、
多步推理的任务上**彻底 OOD 崩溃**——与解耦机制无关，是训练数据覆盖面太窄的通用副作用。

### 第一次修正尝试：换真实语料(WikiText-103)——问题依旧存在 ⚠️

怀疑根因是训练语料"窄"，改用 `wiki_corpus.py`（WikiText-103真实语料，35万+句子清洗后可用）
重训 lmonly/decoupled（400步/1500句，单卡5090实测约1.6s/step，两模型合计约20分钟）。

**结果：换语料完全没有解决问题**。用简单 chat 问答（"What is 2+2?"）快速诊断：
base 模型至少还能勉强作答（`"is equals.."`），而 lmonly 依然直接崩溃成重复token
（`"222222 and"`）。GSM8K 完整评估同样几乎全崩溃。

**真正的根因诊断**：Dream-7B-Instruct 是靠 chat 模板（system/user/assistant）驱动指令跟随的
模型。用"裸陈述句（无chat结构）+ 随机位置mask做infilling"训练，本质是在教模型做"填空"，
这与 chat/指令跟随行为在格式上直接冲突——哪怕语料内容换成了维基百科式的多样真实文本，
只要训练格式不是 chat 结构，几百步 LoRA 就会破坏模型的 chat 模板生成能力。
**问题不是语料"窄"，是训练格式与评估格式（chat结构+多轮）不匹配导致的灾难性遗忘。**

### 第二次修正：用 GSM8K 训练集构造 chat 格式语料（格式对齐评估）

改用 GSM8K **训练集**（7473题，与测试集完全不重叠）构造完整 chat 序列
`system + user(question) + assistant(推理链+#### answer)`，只在 assistant 回复片段内做
随机mask/解耦训练，prompt 部分（system+user+response前缀）永远 keep 不动——训练格式与
评估格式严格对齐（实现见 `experiments/proposition_c/gsm8k_corpus.py`）。

**单步耗时明显变慢**：因序列更长（含完整推理链，截断到320token）且每步~13次全量前向，
实测约 7-15s/step（响应长度方差大），比之前短句训练慢 5-10 倍。

**本机（单卡RTX 5090 Laptop）训练情况**：
- `lmonly`（lam_dec=0）：200步顺利跑完（约30分钟），lm_loss 收敛到 ~0.3，无崩溃迹象。
- `decoupled`（lam_dec=0.5）：训练更慢，本机因工具/资源限制，两次尝试均在训练中途
  （120-140步左右）被意外中断且**未能完整跑完并保存**。中断前的日志显示 lm 收敛正常
  （11.7→0.7），dep 机制指标也在下降（2.26→0.38），方向是对的，但**没有完整结果**。

**判决**：GSM8K chat格式训练方向是对的（格式对齐评估、lmonly训练健康收敛），
但受限于单卡训练速度慢+资源限制，**decoupled 版本未能跑完，因此命题C-v3在GSM8K上的
下游任务验证目前仍缺失完整结果**。已将后续工作迁移到算力更强的 H100 服务器，
详见 `docs/h100_handoff.md`（交接文档，含速度估算与详细 plan）。

用 MDLM 标准式评估（多个 mask 比例 0.15-0.7，只在被 mask 位置算 NLL，避免命中训练时
keep 位置的 identity-shortcut）。语料：Gutenberg《Pride and Prejudice》正文随机采样80段

| 模型 | PPL(masked positions) |
|---|---|
| base(冻结未训练) | 1026.8 |
| lora_only(纯LM微调) | 522.6 |
| decoupled(精准解耦) | **273.8** |

**结论**：
- **未观察到解耦损害语言能力**——PPL 没有变差，反而三者中最好。排除了命题C-v2式的最坏情况。
- 但**不能简单归因为"解耦机制生效"**：两个微调模型都比base好，说明 LoRA 训练本身

## 总体结论（截至本轮）

1. **commit-readiness 是独立于 confidence 的真实量**（Exp1-3, A 成立）。
2. **它主要是 state-local 可判定的**（hidden 单独 R²≈0.34，加全局信息仅+0.06），
   **除生成最早期**（nrev≤2 时存在真实 path-dependence，Δlocal≈0.09）。
3. **它编码在浅层最强**（第2层 R²≈0.38，U 形分布）——像浅层表面可预测性统计。
4. **模型自发生成是"边界锚点优先"**（L2R corr≈0.62；先定首尾），且**锚点是位置性而非语义性**（Exp8 反证 information-unit）。
5. **学到的 certifier 在并行 commit 上显著优于 confidence**（50%并行时错误率降~80%，8/8种子稳健）。
6. **AR 退化的机制 = confidence 门控**：高 confidence 几乎只在上下文充分后出现，
   从而强制串行；certifier 能在低 confidence 时识别 stable 位置，是打破串行的关键（Exp9）。
7. **命题C-v3(精准解耦)机制成立**：训练能主动增加条件独立结构（强依赖69%→28%，可并行子集翻倍），
   有严格对照排除微调副作用，落在DEMASK/PUNT等inference-only方法的盲区。
8. **命题C-v3下游任务验证未完成**：GSM8K真实评估暴露训练语料格式（需chat结构对齐）的关键教训，
   decoupled版本在GSM8K chat语料上的训练受限于单卡资源未能跑完，已迁移H100继续（见`docs/h100_handoff.md`）。

**综合**：certifier / representation 路线在这个 MDLM 上有扎实经验支撑，命题C-v3的机制结果
是目前最有价值、最接近文献真空区的发现，但因果链后半段（降依赖→更好下游表现）仍待H100上坐实。

## 已知局限（不夸大）
